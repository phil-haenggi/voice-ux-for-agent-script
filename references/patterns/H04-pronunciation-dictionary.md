# Pronunciation dictionary

## Why it matters

TTS engines mispronounce brand names, product names, medical specialities, and domain jargon at predictable rates. "Nuffield" gets read as "Nu-field" or "Nuh-feeld". "Orthopaedics" mispronounces in en-US engines defaulting to en-GB content. Hospital town names pronounce inconsistently ("Newcastle" stress varies).

A pronunciation dictionary is the platform-level fix: a list of `term → IPA` (or CMU/ARPAbet) mappings the TTS engine consults before synthesis.

This pattern is **out-of-`.agent`** — the dictionary lives in voice configuration, not in the script. The skill generates a starter list of candidates from terms used in the agent and outputs it as a side artifact (`pronunciation-dictionary.md`).

## Detection

Source uses recurring proper nouns or domain terms:
- Brand names ("Nuffield", "Bupa", "AXA")
- Hospital / location names ("Newcastle", "Guildford", "Highgate")
- Medical specialities ("Orthopaedics", "Otolaryngology", "Cardio-Thoracic")
- Insurer names
- Foreign-origin terms or names

The skill scans user-visible content + topic instructions and proposes candidates.

## Voice example (after) — recommended

**Generate a starter pronunciation dictionary, output as side artifact** (couples with `templates/pronunciation-dictionary.md`):

```markdown
# Pronunciation Dictionary — starter list (verify in voice preview)

| Term | IPA | Notes |
|---|---|---|
| Nuffield | ˈnʌfiːld | Brand name. Default en-GB engine often gets wrong. |
| Orthopaedics | ɔːθəˈpiːdɪks | Stress on "pae". |
| Cardio-Thoracic | ˌkɑːdɪoʊθəˈræsɪk | Hyphenated; some engines split as two words. |
| Newcastle (upon Tyne) | ˈnjuːkɑːsl | Stress on "New", not "cas". |
| Bupa | ˈbuːpə | Insurance brand. |

⚠️ All entries marked unverified — load into Agentforce Builder → Voice Settings → Pronunciation Dictionary, then test in voice preview. Adjust IPA if engine pronunciation still wrong.
```

The dictionary entries are added at the org level (Voice Settings → Pronunciation Dictionary), not in the `.agent` file.

## Voice example — alternatives

**SSML `<phoneme>` overrides** inline in instructions: works without org-level dictionary, but the LLM has to emit consistently:

```
| When mentioning the company, use:
| "<phoneme alphabet='ipa' ph='ˈnʌfiːld'>Nuffield</phoneme> Health"
```

Brittle — the LLM occasionally drops the phoneme tag. Org-level dictionary is more reliable.

**Skip if all terms have standard pronunciations.** If the agent's vocabulary is everyday English with no brand names or jargon, no dictionary needed.

## Anti-patterns

- **Setting IPA without testing.** The engine's interpretation of IPA varies. Always verify in voice preview, not by reading the IPA back.
- **Adding every English word.** The dictionary is for non-default pronunciations only. Adding everything bloats config and slows TTS.
- **Forgetting to deploy the dictionary alongside the agent.** The skill outputs `pronunciation-dictionary.md` as a starter; humans must load it into the org's voice settings. Add to Layer E review checklist.
- **Hard-coding ARPAbet on an IPA-only engine** (or vice versa). Verify the engine's accepted format before generating.

## Where it lives

- Org-level voice configuration (Agentforce Builder → Voice Settings → Pronunciation Dictionary).
- Side artifact output: `pronunciation-dictionary.md` (generated by Phase 5/7).

## Mis-applications (optimize mode)

- **Pattern: dictionary exists but only covers brand names, not specialities.** Add the speciality list (these mispronounce often).
- **Pattern: dictionary entries are hard-coded for one TTS engine and the org switched engines.** IPA is portable but verify; CMU/ARPAbet may not be.
- **Pattern: dictionary loaded once and never updated as new terms enter the agent.** Add to a periodic review process.
