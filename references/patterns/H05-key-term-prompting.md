# Key-term prompting (STT bias)

## Why it matters

STT engines have generic vocabularies. Domain-specific terms — insurer names, hospital names, consultant names, medical specialities — get mistranscribed at high rates because the model's prior probability for these tokens is low. "AXA" becomes "axe a", "Aviva" becomes "a viva", "Cardio-Thoracic" becomes "cardio thora SIC".

The platform-level fix is **key-term prompting** (also called context biasing or hot-words): a list of expected terms passed to the STT engine to raise their prior probability during recognition. Modern STT engines (Google, ElevenLabs, Polly with custom vocabulary, etc.) support this via different APIs.

Like H04 (pronunciation dictionary), this is **out-of-`.agent`** configuration. The skill generates a starter list as a side artifact.

## Detection

Source has topics that collect or recognise:
- Insurer names (`find_insurer`)
- Consultant names (`find_closest_consultant_names`)
- Hospital names (from `Get_Hospitals_within_Specified_Distance` results)
- Medical specialities (the supported-list mapping in E04)
- Postcodes (the alphanumeric structure benefits from biasing toward UK-format)

## Voice example — alternatives

**Inline biasing per topic:** some platforms allow per-action / per-topic biasing. Useful if the term list varies (e.g., bias toward speciality names only when in the speciality-collection turn).

**Confidence-driven explicit confirmation as fallback:** if biasing isn't available or doesn't fix recognition, accept that some terms will mishear and design Tier-2 repair for them ("Sorry, did you mean AXA, A-X-A, or something similar?"). Combines with D-category spell-confirm patterns.

**Org-level vocabulary lists:** for large term sets (full hospital list across the country), load into the org's persistent voice vocab rather than per-call biasing.

## Anti-patterns

- **Listing 1000+ terms.** STT engines have practical limits (varies by vendor — some hard-cap at 100-500 hot-words). Prioritise the top recurring terms.
- **Including common English words.** Biases the engine away from natural speech. Domain-specific only.
- **Generating the list once and never updating.** New consultants, new insurers, new hospitals — the list ages. Add to a periodic refresh process.
- **Ignoring postcode biasing.** Postcodes mishear constantly; biasing to UK format dramatically improves capture.

## Where it lives

- Org-level voice configuration (Voice Settings → Key-Term Prompting / context biasing / custom vocabulary, depending on platform).
- Side artifact output: `key-term-prompting.md` (generated by Phase 5/7).

## Mis-applications (optimize mode)

- **Pattern: list covers insurers but not specialities.** Specialities mishear at higher rates than insurers (more syllables, more medical-Latin). Add.
- **Pattern: list copied verbatim from another org.** Locale and customer base differ. Regenerate from this agent's actual vocabulary.
- **Pattern: list deployed but not verified with sample utterances.** Recognition still fails for terms that aren't actually being biased correctly. Test before relying.
